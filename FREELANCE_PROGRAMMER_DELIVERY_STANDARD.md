# GovScout.pro Freelance Programmer Delivery Standard

Effective date: 2026-05-26  
Relationship: External programmer delivery rules for GovScout.pro work

## Direct Rule

You are a freelance programmer implementing work for GovScout.pro.

You do not set product policy. You do not decide pricing. You do not approve security risk. You do not declare launch readiness. You do not replace working functionality with reports, placeholders, demo behavior, or unsupported claims.

You build the assigned change, prove it works, disclose limitations, and return it for GovScout acceptance.

## Work Order Format

Every programming assignment must contain:

| Field | Required Detail |
| --- | --- |
| Work order ID | Unique GovScout identifier linked to Launch Control |
| Priority | `P0`, `P1`, `P2`, or `P3` |
| Customer failure | What is broken or what customer outcome is missing |
| Required behavior | Exact behavior the delivered system must exhibit |
| Scope | Files, services, database objects, pages, functions, or integrations included |
| Exclusions | What must not be changed without GovScout.pro written approval |
| Security rules | Authentication, authorization, data, secret, and payment requirements |
| Acceptance tests | Tests and user workflows that must pass |
| Evidence due | Code changes, test output, screenshots where relevant, configuration/migration notes |
| Delivery deadline | Agreed date/time and escalation route if blocked |
| GovScout owner | Internal person/team accepting or rejecting the delivery |

## Freelancer Delivery Requirements

For every work order, the programmer must deliver:

1. Root cause or implementation rationale in plain language.
2. List of files and services changed.
3. Database migrations, environment variables, third-party setup, and deployment effects.
4. Automated tests added or changed and their results.
5. Manual verification steps for customer-visible behavior.
6. Security and payment impact statement when relevant.
7. Known limitations or unresolved dependencies.
8. Rollback instructions for risky deployments.

## Automatic Rejection

GovScout rejects a delivery immediately if it:

- Does not run or does not satisfy acceptance tests.
- Fixes only the visible symptom while the broken or insecure workflow remains.
- Trusts caller-provided identity, payment, entitlement, or ownership data without verification.
- Exposes credentials, customer data, private uploads, session tokens, or administrative controls.
- Introduces fake data, demo fallback, generic Marcus behavior, or false marketing claims as though they are production behavior.
- Changes pricing, entitlements, checkout provider, or public claims without written GovScout.pro work-order approval.
- Requests completion or payment without usable proof.

## Delivery Statuses

| Status | Meaning |
| --- | --- |
| `ASSIGNED` | GovScout issued a written work order |
| `IN DEVELOPMENT` | Programmer acknowledged scope and is implementing |
| `DELIVERED FOR REVIEW` | Programmer submitted code and required evidence |
| `REJECTED - CORRECTION REQUIRED` | GovScout found failure, missing proof, or out-of-scope work |
| `ACCEPTED LOCAL` | GovScout verified the delivered build outside production |
| `DEPLOYED - RETEST REQUIRED` | Authorized deployment occurred; production evidence is pending |
| `VERIFIED CLOSED` | GovScout QA and Executive Launch Command accepted production behavior |

## Production Access Rule

Freelance programmers do not deploy, change live configuration, rotate secrets, edit billing products, modify customer data, or access production logs containing customer information unless GovScout.pro provides written authorization for that specific operation after any required owner `YES` decision is recorded.

Any production-authorized work must be logged in Launch Control and followed by GovScout-owned retesting and approval.
